# 智能体工作流操作手册

> **版本**: 1.0.0
> **适用范围**: 业务分析师、策略工程师、系统运维人员
> **最后更新**: 2026-02-17

## 1. 简介 (Introduction)

本手册旨在指导用户如何设计、配置和运行基于“智能体 (Agent)”的工作流。
智能体工作流是一种结合了大语言模型 (LLM) 推理能力与传统工作流流程编排的新型自动化模式。它允许用户将复杂的业务分析任务拆解为由于多个智能体协作完成的流程，支持单智能体推理、多智能体辩论、裁判裁决等多种模式。

---

## 2. 核心概念 (Core Concepts)

在开始设计工作流之前，需要理解以下核心概念：

### 2.1 智能体配置 (Agent Profile)
智能体配置定义了一个智能体的“人设”和“能力边界”。
*   **角色类型 (Role Type)**: 智能体扮演的角色，如“多头分析师”、“空头分析师”、“风险控制员”。
*   **模型配置 (Model Config)**: 指定使用的 LLM 模型（如 Gemini Pro 1.5, GPT-4o）及参数（Temperature, Max Tokens）。
*   **关联提示词 (Prompt Code)**: 绑定的提示词模板编码。

### 2.2 提示词模板 (Prompt Template)
提示词模板决定了智能体如何思考和输出。
*   **系统提示词 (System Prompt)**: 定义智能体的长期记忆、行为准则和身份设定。
*   **用户提示词 (User Prompt)**: 包含动态变量（如 `{{input.price}}`），每次执行时会根据上下文填充数据。
*   **输出格式 (Output Format)**: 通常设置为 `JSON`，配合 JSON Schema 确保输出结构化数据。

### 2.3 工作流节点 (Workflow Node)
在工作流画布中，智能体以“节点”的形式存在：
*   **智能体调用 (Agent Call)**: 执行单次智能体推理。
*   **辩论轮次 (Debate Round)**: 组织多个智能体进行多轮辩论。
*   **裁判智能体 (Judge Agent)**: 对辩论结果进行汇总和裁决。

## 3. 设计指南 (Design Guide)

### 3.1 单智能体模式 (Single Agent Mode)
适用于独立的分析任务，例如：
*   **市场日报摘要**: 输入长文本，输出结构化摘要。
*   **数据清洗**: 识别非结构化文本中的关键实体（价格、地点）。
*   **单点预测**: 基于历史数据给出涨跌判断。

**设计步骤**:
1.  在“智能体中心”创建一个专注于特定任务的 Agent Profile。
2.  在工作流中拖入 `Agent Call` 节点。
3.  选择对应的 Agent Profile。
4.  配置输入变量映射。

### 3.2 多智能体辩论模式 (Debate Mode)
适用于复杂、模糊或存在对抗性观点的场景，例如：
*   **后市行情研判**: 多头 vs 空头。
*   **重大风险评估**: 业务方 vs 风控方。

**设计步骤**:
1.  准备至少 2 个即便观点对立的 Agent Profile（例如 Bullish Agent 和 Bearish Agent）。
2.  在工作流中拖入 `Context Builder` 节点（构建辩论背景）。
3.  连接 `Debate Round` 节点，配置参与者和轮次。
4.  连接 `Judge Agent` 节点，对辩论结果进行裁决。

### 3.3 提示词工程 (Prompt Engineering)

在编写 `User Prompt` 时，支持使用 Mustache 语法 `{{variable}}` 引用上下文数据：

*   **引用上游输入**: `{{input.some_field}}`
*   **引用全局参数**: `{{params.threshold}}`
*   **引用完整上下文**: `{{context}}` (慎用，可能导致 Token 溢出)

**示例**:
```markdown
请分析以下市场数据：
品种: {{input.commodity}}
当前价格: {{input.current_price}}
历史波动率: {{input.volatility}}

请给出未来 3 天的走势预测（看涨/看跌/震荡）。
```

## 4. 配置手册 (Configuration Manual)

### 4.1 创建智能体 (Agent Center)
1.  进入 **Agent Center (智能体中心)**。
2.  点击 **Create Agent**。
3.  填写基本信息：
    *   **Agent Code**: 唯一标识，如 `AGENT_CORN_BULL`。
    *   **Role Type**: 选择角色标签。
    *   **Model Config**: 选择默认模型（支持在节点级别覆盖）。
4.  编写提示词模板：
    *   推荐使用 **Few-Shot (少样本学习)**，在 `Few-Shot Examples` 区域添加 3-5 个高质量的输入/输出对，能显著提高稳定性。

### 4.2 节点参数配置详解

#### Agent Call 节点
| 参数 | 说明 | 推荐值 |
| :--- | :--- | :--- |
| **Agent Profile** | 选择要执行的智能体 | 必选 |
| **Model Override** | 覆盖默认模型配置 | 视任务复杂度而定 |
| **Temperature** | 随机性 (0-1)，越低越稳定 | 0.1 (数据提取) / 0.7 (创意生成) |
| **Variables** | 自定义局部变量 | 按需配置 |

#### Debate Round 节点
| 参数 | 说明 | 推荐值 |
| :--- | :--- | :--- |
| **Max Rounds** | 最大辩论轮次 | 2-3 轮 |
| **Participants** | 参与辩论的智能体列表 | 2-4 个 |
| **Consensus Threshold** | 共识阈值 (0-1)，达到即提前结束 | 0.8 |
| **Judge Policy** | 裁判策略 (Weighted/majority/JudgeAgent) | Weighted (加权平均) |

### 4.3 输出校验配置 (Guardrails)
为了防止大模型产生幻觉，必须配置输出校验：

1.  **Output Format**: 必须设置为 `JSON`。
2.  **Output Schema**: 必须提供 JSON Schema 代码。
3.  **Guardrails 规则**:
    *   `requiredFields`: 必填字段列表。
    *   `minConfidence`: 最低置信度要求（如 0.6）。
    *   `forbiddenPatterns`: 禁止出现的词汇。

## 5. 执行原理与监控 (Execution & Monitoring)

### 5.1 执行流程解析
当工作流执行到 Agent 节点时：
1.  **Context Loading**: 系统加载上游节点的输出 `outputSnapshot`。
2.  **Prompt Interpolation**: 将数据注入 `User Prompt` 模板。
3.  **LLM Invocation**: 调用配置的模型 API (Google/OpenAI)。
    *   *自动重试*: 如果调用失败，系统会根据 `Retry Policy` 自动重试（默认 3 次）。
4.  **Parse & Validate**: 系统解析返回的 JSON，并执行 Schema 校验。
    *   *校验失败*: 节点标记为 `FAILED`，并记录错误原因 (如 "缺少必填字段 confidence")。

### 5.2 监控与日志
在 **Workflow Execution Detail** 页面：
*   **Run Log**: 查看每个节点的输入、输出和耗时。
*   **Trace View**: 对于辩论节点，可以查看每一轮的详细发言记录。
*   **Cost**: 查看 Token 消耗量（需配置计费模块）。

## 6. 故障排查 (Troubleshooting)

### 6.1 常见错误代码

| 错误信息 | 原因分析 | 解决方案 |
| :--- | :--- | :--- |
| `Agent[XXX] 调用鉴权失败` | API Key 无效或过期 | 检查 `AIModelConfig` 中的 API Key 配置 |
| `JSON 解析失败` | 模型输出了非 JSON 格式文本 | 1. 降低 Temperature<br>2. 优化 System Prompt 强调 JSON 格式<br>3. 增加 Few-Shot 示例 |
| `缺少必填输出字段: xxx` | 模型遗漏了字段 | 检查 Prompt 是否明确要求了该字段；检查 Output Schema 定义 |
| `置信度 x 低于阈值 y` | 模型对结果不自信 | 这是一个业务逻辑阻断，通常意味着需人工介入；或尝试提供更多上下文信息 |

### 6.2 调试技巧
1.  **使用 "Test Run"**: 在 Agent Center 中可以单独测试 Prompt，不依赖工作流。
2.  **查看 "Raw Response"**: 在执行日志中查看 LLM 的原始返回内容，往往能发现 Prompt 的歧义点。
3.  **分步调试**: 将长 Prompt 拆分为多个简单的 Agent 串联执行。
