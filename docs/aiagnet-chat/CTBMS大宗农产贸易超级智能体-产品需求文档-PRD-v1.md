## 0. 执行摘要

CTBMS 将建设一个面向大宗农产贸易的超级智能体，核心不是“更会聊天”，而是“更懂业务、可证据化、可执行”。

产品通过统一数据底座（内部业务数据 + 公开资讯 + 期货 API + 天气 API + 物流 API），支撑两条主链：

1. 智能体工作流链：数据汇聚 -> 分析推理 -> 风控校验 -> 交付执行。
2. AI 对话链：自然语言提问 -> 计划生成 -> 工具调用 -> 证据化回答。

最终目标：把“信息检索 + 研判 + 预警 + 复盘 + 协同交付”整合为一个可持续扩展、可审计、可复用的产品能力中心。

## 1. 背景与 Why Now

### 1.1 业务背景

大宗农产贸易决策依赖多源信息：

- 内部经营数据（库存、采购、销售、履约、利润）。
- 市场与公开信息（新闻、研报、政策、舆情）。
- 期货与衍生数据（行情、基差、持仓、期限结构）。
- 外部影响因子（天气、物流、港口、运费、突发事件）。

当前问题在于数据来源分散、口径不统一、结论不可复现，导致决策效率与信任成本较高。

### 1.2 为什么是现在

1. 已具备工作流执行与对话式智能体能力，缺口集中在“数据底座产品化”。
2. 市场波动频繁，天气与物流扰动对价格和履约影响显著上升。
3. 业务对“可追溯结论”和“可复盘建议”需求已从加分项变为刚需。
4. 新 API（天气、物流等）接入窗口成熟，适合建立统一接入标准。

## 2. 产品愿景与定位

### 2.1 产品愿景

打造农产大宗贸易领域的“证据驱动超级智能体”，让一线业务人员可以通过对话或模板快速获得可执行建议，并清楚知道建议来自哪些数据与规则。

### 2.2 产品定位

- 不是通用聊天机器人。
- 是面向交易与经营场景的决策操作系统。
- 回答必须“可验证、可追溯、可复盘、可治理”。

## 3. 用户与核心场景

| 用户角色        | 主要诉求               | 高频任务                     | 关键输出            |
| --------------- | ---------------------- | ---------------------------- | ------------------- |
| 交易员/经营人员 | 快速判断行情与动作     | 当日策略、风险边界、对冲建议 | 可执行动作清单      |
| 研究员/分析师   | 多源信息归因与趋势研判 | 周报、月报、专题分析         | 结构化报告 + 证据   |
| 风控人员        | 降低误判和过度暴露     | 异动预警、冲突识别、阈值告警 | 风险等级 + 处置建议 |
| 采购与物流团队  | 提前识别履约与运输风险 | 路径扰动、天气冲击评估       | 供应链影响评估      |
| 管理层          | 获得可追责的决策依据   | 经营复盘、策略评估           | 结论摘要 + 审计链   |

核心场景：

1. 周度市场复盘与未来 2-12 周趋势展望。
2. 异常预警（价格、基差、库存、物流延迟、天气冲击）。
3. 对话发起策略问题并一键生成报告与订阅。
4. 历史策略回测与“结论变化原因”解释。

## 4. 产品目标与非目标

### 4.1 产品目标

1. 建立统一数据底座，支持多源数据标准化接入与治理。
2. 让工作流与对话都基于同一套数据契约和证据体系。
3. 提供模板化场景，覆盖研判、预警、复盘、订阅四类高频任务。
4. 实现“新 API 接入后可被智能体快速复用”的扩展能力。

### 4.2 非目标（MVP）

1. 不做自动下单或全自动交易执行。
2. 不做全市场全品类一次性覆盖，优先玉米/豆粕/小麦等重点品类。
3. 不替代人工风控审批责任。

| 数据新鲜度达标率 | 使用数据在 TTL 内的比例  | >= 98%   |
| 流程成功率       | 工作流端到端成功执行比例 | >= 99%   |

## 6. 产品范围（MVP）

### 6.1 模块范围

1. 数据接入中心（Connector Hub）。
2. 语义与指标中心（Semantic & Metrics Layer）。
3. 证据引擎（Evidence Engine）。
4. 智能体工作流中心（Workflow Runtime + Templates）。
5. AI 对话副驾（Copilot）。
6. 治理与观测中心（Quality, Audit, Guardrail）。

### 6.2 场景范围

1. 市场周报自动生成。
2. 价格与风险异动预警。
3. 天气与物流扰动影响评估。
4. 策略回测与订阅交付。

### 6.3 实施边界与复用策略

1. 后端能力以现有 `agent-conversations` 与 `workflow-execution` 为核心，不重写平行会话域。
2. 如需品牌化入口，`/super-agent/*` 仅作为 BFF 路由别名，不新增重复 Service/Repository。
3. 数据底座采用“存量标准化 + 增量接入”同步推进，禁止只接新源不改旧口径。
4. 工作流与对话新能力只读标准层/指标层，逐步下线对原始源表的直接耦合。

### 6.4 数据冗余控制与存量改造策略

1. 采用“单写入、统一读取”原则：采集链路保持单主写入，分析链路统一读标准层。
2. 采用“先映射、后改造、再收敛”路径：先标准适配，再做定向结构改造，最后收敛 legacy 读路径。
3. 迁移期间默认双跑对账，未达阈值前不得切断 legacy 读取能力。
4. 新接入数据源必须先通过标准层与质量门禁，不允许绕过标准层直连工作流。
5. 仅当以下条件满足时才做深度重构：
   - 关键性能瓶颈长期无法通过适配层解决。
   - 双口径导致业务误判风险不可接受。
   - 维护成本持续高于重构成本。

| 改造级别 | 适用条件                         | 预估成本 | 风险 | 默认策略     |
| -------- | -------------------------------- | -------- | ---- | ------------ |
| 映射层   | 需要快速统一口径、保持采集稳定   | 低       | 低   | 默认起步     |
| 轻改造层 | 需要补齐质量/时效/置信度治理能力 | 中       | 中   | 推荐并行推进 |
| 深重构层 | 结构性瓶颈显著、收益明确         | 高       | 高   | 仅按需执行   |

## 7. 产品方案总览

### 7.1 七层能力架构

1. 数据源层：内部系统、公开信息、期货、天气、物流等。
2. 接入层：统一 Connector 契约、限流、缓存、重试。
3. 标准层：字段映射、单位换算、时区统一、主数据对齐。
4. 特征层：供需、库存、运费、天气扰动等业务特征。
5. 推理层：工作流节点执行 + 智能体协作。
6. 治理层：权限、质量、引用、风控门禁。
7. 交付层：对话输出、报告导出、订阅推送、审计回放。

### 7.2 两条执行主链

1. 对话链：问题 -> 意图识别 -> 槽位补全 -> 计划确认 -> 工具执行 -> 证据化回答。
2. 工作流链：模板/策略 -> 数据就绪校验 -> 编排执行 -> 风险门禁 -> 结果交付。

## 8. 数据底座设计（核心）

### 8.1 数据域规划

| 数据域       | 典型内容                     | 更新频率     | 主要用途       |
| ------------ | ---------------------------- | ------------ | -------------- |
| 内部经营数据 | 采购、销售、库存、履约、利润 | 分钟/小时/日 | 经营决策主数据 |
| 市场公开信息 | 新闻、研报、政策、舆情       | 分钟/小时    | 事件驱动解释   |
| 期货 API     | 行情、持仓、基差、期限结构   | 秒/分钟      | 价格与风险研判 |
| 天气 API     | 温度、降水、灾害预警         | 小时/日      | 产量与运输影响 |
| 物流 API     | 运价、线路时效、拥堵状态     | 小时         | 履约与成本评估 |
| 扩展数据域   | 港口、船运、宏观指标         | 日/周        | 中长期研判增强 |

### 8.2 Connector 统一契约

每个数据源接入必须配置以下字段：

1. `connectorId`、`sourceType`、`owner`。
2. `authConfig`（密钥、token、轮换策略）。
3. `requestSchema`、`responseSchema`（强类型）。
4. `retryPolicy`、`timeoutPolicy`、`rateLimitPolicy`。
5. `cachePolicy`（TTL、回源策略、降级策略）。
6. `freshnessSla`（时效等级与超时处理）。
7. `qualityRules`（完整性、一致性、异常阈值）。
8. `permissionScope`（可访问组织与字段范围）。

### 8.3 语义标准与指标字典

统一主数据实体：

- `commodity`（品类）
- `region`（产区/销区）
- `warehouse`（仓库/港口）
- `route`（物流线路）
- `contract`（贸易合同）
- `futuresSymbol`（期货合约）

MVP 核心指标：

1. 现货价格、期货价格、基差。
2. 库存天数、库存环比、库存同比。
3. 采购成本、物流成本、毛利率。
4. 运价指数、运输时效偏差。
5. 天气扰动指数、供应风险指数。

### 8.4 数据质量评分与门禁

质量评分（示例）：

`qualityScore = 完整性*0.35 + 时效性*0.30 + 一致性*0.20 + 异常稳定性*0.15`

门禁策略：

1. `qualityScore < 0.7`：禁止输出强结论，降级为观察建议。
2. 核心字段缺失：阻断流程并提示补充来源。
3. 多源冲突超阈值：输出冲突说明与影响范围。

### 8.5 证据包（Evidence Bundle）

事实型输出必须绑定证据：

```json
{
  "claim": "未来两周玉米现货波动风险上升",
  "evidence": [
    {
      "source": "futures_api",
      "metric": "basis",
      "time": "2026-02-26T09:30:00+08:00",
      "value": -45,
      "confidence": 0.91
    },
    {
      "source": "weather_api",
      "metric": "rainfall_anomaly",
      "time": "2026-02-26T08:00:00+08:00",
      "value": 1.8,
      "confidence": 0.84
    }
  ],
  "conflicts": [],
  "freshness": "within_ttl"
}
```

### 8.6 存量底座可用性评估（当前结论）

当前存量数据源已经具备成为优秀底座的基础条件，但需要完成标准化收敛后才能稳定支撑“超级智能体产品级能力”。

| 存量资产                                                      | 当前状态                     | 底座价值 | 主要改造点                                               |
| ------------------------------------------------------------- | ---------------------------- | -------- | -------------------------------------------------------- |
| 采集点价格连续性数据（`PriceData`）                           | 已规模化沉淀，具备时间连续性 | 高       | 统一品类/区域编码、单位语义、质量分与时效标签            |
| 知识库日报/周报/月报/研报（`KnowledgeItem`/`ResearchReport`） | 已形成文本与结构化摘要       | 高       | 统一事件类型、影响方向、引用片段和发布时间语义           |
| B 类情报库（`MarketIntel` + 事件/洞察）                       | 已入库并有评分字段           | 高       | 将评分体系映射到标准 `quality/confidence` 并纳入冲突仲裁 |
| 期货快照与衍生特征（`FuturesQuoteSnapshot`）                  | 已可用于盘面分析             | 中高     | 与现货/情报统一时间基准和主力合约口径                    |

结论：

1. 你的存量不是“要不要推倒重来”的问题，而是“如何标准化为统一可计算语义层”的问题。
2. 只要完成存量标准化、双跑对账和门禁治理，现有底座可以达到优秀水平并支撑后续扩展。

## 9. 智能体与工作流设计

### 9.1 智能体角色（MVP）

1. 行情智能体：价格、基差、期限结构分析。
2. 基本面智能体：供需、库存、政策归因。
3. 天气物流智能体：天气冲击与运输摩擦评估。
4. 风控智能体：风险分层、阈值告警、边界控制。
5. 裁判智能体：冲突收敛与最终建议等级输出。

### 9.2 工作流模板（MVP）

1. `周度复盘模板`：价格 + 事件 + 风险 + 建议。
2. `异动预警模板`：实时监控 + 阈值触发 + 通知。
3. `天气物流评估模板`：扰动 -> 影响 -> 应对方案。
4. `策略回测模板`：建议复盘 + 指标评估 + 结论迭代。

### 9.3 执行门禁

1. 数据门禁：时效与质量达标才可继续。
2. 权限门禁：越权数据访问直接拒绝。
3. 风控门禁：高风险建议默认附带人工确认提示。
4. 成本门禁：超预算自动降级检索深度或轮次。

## 10. 对话体验设计

### 10.1 对话状态机

1. `INTENT_CAPTURE`：识别任务类型。
2. `SLOT_FILLING`：补齐时间、区域、品类、指标、输出方式。
3. `PLAN_PREVIEW`：展示执行计划与成本预估。
4. `USER_CONFIRM`：用户确认或改写关键参数。
5. `EXECUTING`：工具调用与工作流执行。
6. `RESULT_DELIVERY`：输出结论、证据、报告、订阅入口。

### 10.2 输出结构标准

每次结果固定四段：

1. 结论摘要（可执行）。
2. 证据依据（来源与时间戳）。
3. 风险与不确定性。
4. 下一步建议（可选动作）。

## 13. 里程碑与发布计划

### M1（第 1-4 周）：数据底座最小闭环

3. 质量评分与 freshness 门禁上线。

1. 天气 API、物流 API 接入并标准化。

## 14. 风险与应对

| 风险                 | 影响           | 对策                                   |
| -------------------- | -------------- | -------------------------------------- |
| 外部 API 不稳定      | 执行失败率上升 | 缓存、重试、降级、供应商冗余           |
| 多源数据冲突         | 结论可信度下降 | 冲突仲裁规则 + 裁判智能体 + 置信度标注 |
| 权限边界不清         | 合规风险       | RBAC + 数据域隔离 + 审计追踪           |
| 用户过度依赖自动建议 | 决策风险       | 高风险建议强制风险声明与人工确认       |
| 需求范围膨胀         | 上线延期       | 严格按 MVP 场景与指标验收              |
| 重复建设平行能力     | 维护成本上升   | 复用现有会话/执行链路，新增仅做标准层  |

## 15. 验收标准（UAT）

1. 用户可通过对话完成“提问 -> 执行 -> 证据化结论 -> 导出/订阅”。
2. 系统可在核心场景中同时调用内部、期货、公开、天气、物流数据。
3. 事实型输出具备来源与时间戳，且抽样可复现。
4. 数据质量不达标场景能正确降级并提示原因。
5. 模板场景成功率和时延达到 NFR 目标。

## 16. 依赖与协同

1. 数据侧：外部 API 合同、配额、SLA、成本预算。
2. 业务侧：指标口径确认、风险阈值确认、模板评审。
3. 技术侧：接入网关、缓存组件、任务调度、可观测体系。
4. 合规侧：数据授权、字段脱敏、审计策略。

## 附录 A：MVP 对话示例

用户输入：

> 请分析未来两周东北玉米价格风险，结合天气和物流，给出采购和套保建议，并每周一早上发我一份报告。

系统输出（结构）：

1. 结论：未来两周价格波动风险上升，建议分批采购并提高套保比例。
2. 证据：期货基差、降水异常、线路拥堵指数等来源与时间戳。
3. 风险：天气模型不确定性、物流时效波动、政策扰动。
4. 动作：生成周报模板 + 订阅任务 + 风险阈值告警。
