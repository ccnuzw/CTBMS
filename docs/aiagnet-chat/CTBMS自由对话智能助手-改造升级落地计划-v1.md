## 1. 修订背景与目标

### 1.1 当前问题

1. 页面仍存在“流程感”，普通用户需要理解较多系统概念。
2. 对话发送后回显体验不够即时，用户感知“消息还停留在输入框”。
3. 助手回复中下一步不够明确，用户不知道“还能做什么”。
4. 专业信息（路由、草稿、治理、评分）对普通用户暴露过多，门槛高。

### 1.2 目标重定义（本次修订）

1. 普通用户无感使用：像聊天一样表达需求，直接拿到结果。
2. 前台极简：只保留“对话 + 结果 + 下一步动作”。
3. LLM 主驾驶：以自然语言理解、承接、补全、总结为核心能力。
4. 后台复用与治理继续保留，但默认对普通用户隐藏。

---

## 2. 体验原则（必须遵守）

1. **自由表达优先**：用户不需要按模板提问。
2. **先给结果再追问**：低风险场景优先给可用结果，仅关键缺失时追问一个点。
3. **即时反馈**：用户发送后 200ms 内必须在对话区看到自己的消息气泡。
4. **可选择下一步**：每条助手回复必须包含可执行动作（按钮或建议）。
5. **术语屏蔽**：普通用户不可见 plan/draft/grant/gov/route score 等术语。

---

## 3. 目标形态（普通用户视角）

### 3.1 单屏结构

1. 对话区（主区域）
2. 输入区（自然语言输入）
3. 下一步动作区（回复下方 2-4 个按钮）

### 3.2 助手回复标准结构

1. 结论（一句话）
2. 依据（1-3 条）
3. 下一步建议（按钮化）

### 3.3 下一步按钮标准集

1. 继续追问（默认）
2. 发送报告（邮箱/企微/钉钉/飞书）
3. 定时发送（自然语言）
4. 调整范围（如“改成华北/改成近30天”）

---

## 4. 改造范围

### 4.1 前端（重点）

1. 输入发送链路改造：乐观回显（本地用户气泡 + 助手思考占位）
2. 对话消息卡改造：助手回复支持 `replyOptions` 按钮区
3. 页面信息分层：普通用户默认隐藏专业区，管理员独享“管理工具”
4. 文案体系统一：全量中文自然表达，去工程术语

### 4.2 后端

1. 对话回复结构扩展：增加 `replyOptions` 和 `nextActionHints`
2. LLM 输出规范升级：固定“结论->依据->建议”并产出可点击选项
3. 风险追问策略：仅高风险或关键槽位缺失时追问
4. 保留能力路由与治理，但作为后台能力，不直接暴露给普通用户

### 4.3 类型与协议

1. 扩展 `@packages/types` 对话返回结构
2. 前后端保持兼容：旧字段保留，新字段可选

### 4.4 测试与验收

1. 增加“发送即回显”前端用例
2. 增加“每条助手回复包含下一步选项”接口/e2e 断言
3. 增加“普通用户术语屏蔽”走查清单

---

## 5. 详细落地计划（分阶段）

## Phase A（第 1 周）：交互去流程化（前端主导）

### A1 输入与回显

1. `handleSend` 改造为乐观更新：
   - 发送瞬间追加本地 `USER` 消息气泡
   - 立即清空输入框
   - 同步追加“助手处理中”占位
2. 请求成功后替换占位，失败时显示“重试”动作

### A2 回复动作化

1. 对话消息组件支持 `replyOptions`
2. 每个选项支持两类动作：
   - 填入输入框等待用户发送
   - 直接触发动作（发送报告/创建定时）

### A3 普通用户视图收敛

1. 非管理员隐藏专业模块（治理、路由、草稿、评分）
2. 页面仅保留普通用户主流程元素

---

## Phase B（第 2 周）：LLM 自由对话增强（后端主导）

### B1 对话策略

1. LLM 优先理解自然表达，不要求固定句式
2. 上下文承接增强：支持“按刚才那个/换区域/改周期”
3. 缺槽策略：一次只问一个关键问题并给默认选项

### B2 结果组织

1. 输出强约束为“结论/依据/建议”
2. 生成 2-4 个下一步按钮，避免“纯文字回复后无动作”

### B3 风险控制

1. 低风险：自动执行
2. 高风险：一次确认
3. 所有内部路由与治理信息保持后台可观测

---

## Phase C（第 3 周）：闭环优化与灰度上线

### C1 转化链路优化

1. 结果页动作优先级：继续问 > 发送 > 定时
2. 发送失败提示改为可执行建议（补邮箱/补目标）

### C2 指标埋点

1. 首次结果时长
2. 追问次数
3. 发送转化率
4. 定时任务创建率
5. 用户中断率

### C3 灰度策略

1. 先放量给普通用户 10%
2. 观察 48h 后扩到 30%/60%/100%
3. 异常回退：切回旧回复模式（保留新后端能力）

---

## 6. 任务拆解（可直接执行）

### 6.1 Web 任务

1. `AgentCopilotPage`：实现本地乐观气泡与占位消息
2. 新增消息动作区组件：`replyOptions` 渲染与点击处理
3. 普通用户隐藏专业区，管理员入口保留
4. 全量文案走查并替换术语

### 6.2 API 任务

1. `createTurn` 返回结构加入 `replyOptions`
2. LLM 生成层加入“下一步动作建议”生成逻辑
3. 风险追问策略统一（高风险确认、低风险自动推进）
4. 维持现有能力复用、路由日志、治理机制

### 6.3 Types 任务

1. 更新 `ConversationTurn`/`CreateTurnResponse` 相关 schema
2. 增加 `ReplyOption` 类型定义

### 6.4 测试任务

1. 前端：发送后即时回显测试
2. API：回复包含 `replyOptions` 的契约测试
3. E2E：普通用户从提问到发送报告的完整链路
4. 文案验收：首屏术语屏蔽检查

## 7. 里程碑与交付物

### M1（Week 1）

1. 前端自由对话基础体验可用
2. 普通用户默认视图收敛完成

### M2（Week 2）

1. LLM 自由对话策略上线灰度
2. 回复动作化（replyOptions）全链路可用

### M3（Week 3）

1. 指标达标并完成灰度放量
2. 输出上线评审材料（体验、指标、回滚方案）

---

## 8. 验收标准（最终）

1. 用户发送后 200ms 内看到自己的消息气泡。
2. 每条助手回复都有“明确下一步”文字或按钮。
3. 普通用户首屏无专业术语。
4. 新用户 60 秒内可完成“提问 -> 得结果 -> 发送报告”。
5. 低风险场景无需流程式确认即可完成任务。

---

## 9. 风险与回滚

### 9.1 风险

1. LLM 输出不稳定导致选项质量波动。
2. 乐观回显与服务端消息顺序不一致。
3. 文案替换不完整导致术语回流。

### 9.2 回滚预案

1. `replyOptions` 降级为纯文字建议。
2. 关闭乐观回显，回退至服务端回包渲染。
3. 普通用户界面强制隐藏高级模块开关保留。

---

## 10. 结论

本次修订的核心不是继续增加“可配置能力”，而是把现有能力彻底产品化为“自由对话体验”。

对普通用户：只需要自然聊天就能拿到结果与下一步。

对系统：继续保持能力复用、路由治理、审计可追踪，但默认不打扰用户。
